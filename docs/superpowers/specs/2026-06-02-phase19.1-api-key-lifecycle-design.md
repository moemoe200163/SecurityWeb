# Phase 19.1 — API Key Lifecycle (Self-service + Admin)

## Overview

Extend the Phase 19 governance foundation (commit `832596a` — SHA-256 + prefix API key storage) with full key lifecycle management: users can rotate their own key, admins can list/rotate/revoke any user's key. Add explicit `keyRevokedAt` / `keyExpiresAt` semantics so revocation is an authoritative state, not a delete. Tighten `apiKeyAuth` middleware to reject revoked/expired keys with audit logging.

**Out of scope**: key rotation policies / automatic expiry (deferred to 19.4+); key generation during signup (already handled by existing flow); session/JWT-style authentication (project uses API keys only).

## Goals

1. Users can rotate their own API key via self-service endpoint.
2. Admins can list, rotate, and revoke any user's API key.
3. Revoked or expired keys are rejected by `apiKeyAuth` with audit trail.
4. All key operations write to `audit_log` with consistent `action` codes.
5. Backward compatibility: existing `/api/admin/users/:id/api-key` and `/api/admin/retention/cleanup` routes keep working.

## Non-goals

- Automatic key rotation / expiration policies.
- Multi-key per user (one active key at a time).
- Key generation via UI (existing admin endpoint handles initial creation).
- WebAuthn / OAuth / session auth (project remains API-key-only).

## Schema Migration

**File**: `backend/prisma/migrations/<TIMESTAMP>_api_key_lifecycle/migration.sql` where `<TIMESTAMP>` is the order-prefix generated by `prisma migrate dev` (e.g. `20260602143000`).

```sql
-- Make key columns nullable so revoked users can keep their record
ALTER TABLE users ALTER COLUMN key_prefix DROP NOT NULL;
ALTER TABLE users ALTER COLUMN hashed_key DROP NOT NULL;

-- Add lifecycle timestamps
ALTER TABLE users ADD COLUMN key_created_at TIMESTAMPTZ;
ALTER TABLE users ADD COLUMN key_revoked_at TIMESTAMPTZ;
ALTER TABLE users ADD COLUMN key_expires_at TIMESTAMPTZ;

-- Backfill key_created_at for existing users
UPDATE users SET key_created_at = created_at WHERE key_created_at IS NULL;

-- Index for fast prefix lookup (already exists from Phase 19 commit, verify)
-- CREATE INDEX IF NOT EXISTS idx_users_key_prefix ON users(key_prefix) WHERE key_prefix IS NOT NULL;
```

**Schema** (`backend/prisma/schema.prisma`):

```prisma
model User {
  id           String   @id @default(uuid())
  keyPrefix    String?  @unique @map("key_prefix")
  hashedKey    String?  @unique @map("hashed_key")
  keyCreatedAt DateTime? @map("key_created_at")
  keyRevokedAt DateTime? @map("key_revoked_at")
  keyExpiresAt DateTime? @map("key_expires_at")
  role         String   @default("user")
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
  // ... relations unchanged
}
```

**Important**: `keyPrefix` and `hashedKey` remain `@unique` (Prisma allows multiple NULLs in a unique column in PostgreSQL).

## Service Layer — `backend/src/services/apiKeyService.ts`

All functions write to `audit_log` with consistent shape:

```typescript
{
  actor: <userId>,         // who initiated
  target: <userId>,        // whose key was affected (same as actor for self-service)
  action: 'rotate_key' | 'revoke_key' | 'auth_denied',
  details: { reason?, format? }  // masked via sanitizeAuditDetails
}
```

### Function signatures

```typescript
// Self-service
getMyApiKey(userId: string): Promise<ApiKeyMetadata>
rotateMyApiKey(userId: string): Promise<RotateResult>

// Admin
listAllApiKeys(): Promise<Array<ApiKeyMetadata & { user: { id: string; email: string; role: string } }>>
rotateUserApiKey(targetUserId: string, adminId: string): Promise<RotateResult>
revokeUserApiKey(targetUserId: string, adminId: string, reason?: string): Promise<void>

// Auth helper (used by apiKeyAuth middleware)
isKeyValid(user: User): boolean  // false if null/revoked/expired
```

### Types

```typescript
export interface ApiKeyMetadata {
  prefix: string | null;
  createdAt: string | null;   // ISO
  revokedAt: string | null;
  expiresAt: string | null;
  // lastUsedAt deferred — requires either a User.lastUsedAt column (extra write per request)
  // or auditing every successful auth (noise). Not in 19.1 scope.
}

export interface RotateResult {
  plaintext: string;          // returned ONLY in this response
  metadata: ApiKeyMetadata;
}
```

### Rotation logic (shared by self-service and admin)

```typescript
async function rotate(targetUserId: string, actorId: string): Promise<RotateResult> {
  const { plaintext, prefix, hashed } = generateApiKey();
  const now = new Date();

  await prisma.$transaction([
    prisma.user.update({
      where: { id: targetUserId },
      data: {
        keyPrefix: prefix,
        hashedKey: hashed,
        keyCreatedAt: now,
        keyRevokedAt: null,
        keyExpiresAt: null,
      },
    }),
    prisma.auditLog.create({
      data: {
        userId: actorId,
        action: 'rotate_key',
        resourceType: 'api_key',
        resourceId: targetUserId,
        details: sanitizeAuditDetails({ target: targetUserId }),
      },
    }),
  ]);

  return {
    plaintext,
    metadata: { prefix, createdAt: now.toISOString(), revokedAt: null, expiresAt: null },
  };
}
```

### Revoke logic

```typescript
async function revoke(targetUserId: string, actorId: string, reason?: string): Promise<void> {
  const now = new Date();
  await prisma.$transaction([
    prisma.user.update({
      where: { id: targetUserId },
      data: { keyPrefix: null, hashedKey: null, keyRevokedAt: now },
    }),
    prisma.auditLog.create({
      data: {
        userId: actorId,
        action: 'revoke_key',
        resourceType: 'api_key',
        resourceId: targetUserId,
        details: sanitizeAuditDetails({ target: targetUserId, reason: reason ?? 'admin_action' }),
      },
    }),
  ]);
}
```

## Middleware Changes — `backend/src/middleware/apiKeyAuth.ts`

Current behavior: looks up user by `keyPrefix`, compares `hashedKey` to SHA-256 of input.

**New behavior** (additive, in same order):

1. Extract `X-API-Key` header.
2. Validate format with `isValidKeyFormat` — invalid → 401 `invalid_key_format`, no audit log.
3. Look up user by `keyPrefix` — not found → 401 `unknown_key`, no audit log.
4. If `user.hashedKey` is null OR `user.keyRevokedAt` is non-null → 401 `key_revoked`, audit log `auth_denied` with reason `revoked_key`.
5. If `user.keyExpiresAt` is non-null and `<= now` → 401 `key_expired`, audit log `auth_denied` with reason `expired_key`.
6. Compare `hashApiKey(input)` to `user.hashedKey` — mismatch → 401 `hash_mismatch`, audit log `auth_denied`.
7. Success: attach user to `request.user`, continue.

**Response shape on 401**:

```json
{ "error": "Invalid API key" }
```

(Do not leak which check failed to the client; the audit log captures the internal `code` for ops.)

## Error Handling

| Scenario | HTTP | code (audit) | error | Side effect |
|----------|------|-------------|-------|-------------|
| Missing header | 401 | — | "API key required" | none |
| Bad format | 401 | — | "Invalid API key" | none |
| Prefix not found | 401 | — | "Invalid API key" | none |
| Hash mismatch | 401 | `hash_mismatch` | "Invalid API key" | audit log |
| Revoked key | 401 | `revoked_key` | "Invalid API key" | audit log |
| Expired key | 401 | `expired_key` | "Invalid API key" | audit log |
| Non-admin → `/api/admin/*` | 403 | `forbidden` | "Admin access required" | audit log |
| Self-service w/ wrong userId | 403 | `forbidden` | "Cannot access other user's key" | audit log |
| Rotate DB error | 500 | `rotate_key_failed` | "Failed to rotate key" | audit log |
| Revoke idempotent re-call | 200 | — | (no-op) | no audit log |

Client-side: 401 → existing `ApiKeyRequired` component (no change). 403 → toast "Permission denied". 500 → toast "Operation failed, please retry".

**Audit log noise consideration**: `auth_denied` events fire on every 401. For a public-facing deployment this is noisy; for an internal SOC tool it is desirable. Implement a **per-prefix rate limit** in `apiKeyAuth` middleware: if more than N `auth_denied` events for the same `keyPrefix` in M minutes, throttle to one event per minute. (Implementation detail: keep in-memory `Map<prefix, { count, windowStart }>`; for multi-instance deployment this should move to Redis in a later phase.)
